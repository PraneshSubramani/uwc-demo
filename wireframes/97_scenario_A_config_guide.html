The permissions foundation

Scenario A configures who can see and do what. It is the base layer every other scenario depends on. This guide builds the 4 core profiles (module access), the role hierarchy + data-sharing rules (which records), field-level security (blind review, DOB masking), and the custom action buttons (with visibility conditions) — step by step in Zoho CRM Setup.

The one concept that explains everything: Profiles vs Roles

Zoho splits access into two independent systems. Confusing them is the #1 setup error.

PROFILE = what you can DO

Controls which modules, fields and actions a user can touch (View / Create / Edit / Delete). Profile says "you may edit Volunteers but only view NCs."

e.g. NC Admin profile can create volunteers; Reviewer profile cannot.

ROLE = which records you SEE

Controls data scope via a hierarchy. A user sees their own records + everyone below them in the tree. Role says "you see Brazil's records, not Germany's."

e.g. Ana (NC Admin–Brazil role) sees Brazil; Marcus (NC Admin–Germany) sees Germany — same profile, different role.

So country scoping is a ROLE job, not a profile job

Ana and Marcus share the identical NC Admin profile (same module access) but sit in different roles, so Data Sharing Rules give each only their country's records. Build profiles first (Steps 1–2), then roles (Step 3), then layer field security (Step 4) and actions (Step 5).

Prep — modules & admin access

Setup → Modules and Fields
  1. Confirm you are signed in with an Administrator profile. If you hit "Insufficient privileges", you must be promoted to Administrator before any of this works.
  2. Disable unused stock modules (Vendors, Services, Projects, Sales Inbox, stock Questions) via the module kebab → Disable Module.
  3. Leave enabled: Home · Leads (Applications) · Contacts · Programmes · Campaigns · Tasks · Meetings · Calls · plus the custom modules (NCs, Schools, Place Allocations, Nomination Packs, NC Volunteers, Training Records, MoU Obligations, Governance Bodies/Memberships, Competencies, Mailing Lists, Communication Logs, Consent Records, Safeguarding Cases).

Note

The "Applications" module is the standard Leads module, relabelled. Permission screens will say Leads.

3

Build the role hierarchy + data-sharing rules

Setup → Users and Control → Roles · then Security Control → Data Sharing Settings

The role tree decides record visibility. Build it top-down, then set the org-wide default to Private so the hierarchy actually restricts.

International Office (top — sees everything) ├── NC Admin — Brazil (Ana) │ ├── NC Reviewer Brazil (Clara) │ └── NC Reviewer Brazil (Paulo) ├── NC Admin — Germany (Marcus) │ └── NC Reviewer Germany ├── NC Admin — Kenya (Grace) ├── NC Admin — India · NC Admin — UK └── School Admissions Director (Helen, Erik, Anjali)
Steps
  1. Under Roles, create the tree above — each NC Admin role reports to International Office; each NC Reviewer reports to its NC Admin.
  2. Assign uwc_admin the International Office role + IO Super Admin profile.
  3. Go to Security Control → Data Sharing Settings. Set the default access for record-scoped modules (Leads, Contacts, NC Volunteers, MoU Obligations, Place Allocations) to Private.
  4. With Private default, each role sees only its own records + those below it. Ana (Brazil) cannot see Germany's; Marcus (Germany) cannot see Brazil's — automatically, because they are sibling branches.
  5. If a module should be network-visible (e.g. Programmes), set its default to Public Read Only instead.

This is what enforces NC-level isolation

Profiles gave Ana and Marcus the same module access. The Private default + sibling roles is what makes Ana see only Brazil and Marcus only Germany. Without setting the default to Private, the hierarchy does nothing.

"Own row only" needs a Client Script

Zoho FLS can't scope by row value. Set Reviewer to Read+Write at the field level, then add a Client Script on the Application detail page that blanks peer reviewers' cells unless All_Reviews_Complete = true. Script: zoho-build/client-scripts/blind_review.js. This is what makes the blind-review reveal work in Scenario C.

4b · DOB masking for the Data Protection Lead
  1. Contacts → Field Permissions → Date_of_Birth → for the Data Protection Lead profile choose "View masked" (Zoho's built-in mask) — also apply to Address fields.
4c · Safeguarding sensitivity
  1. Safeguarding Cases → Triage_Notes → IO Super Admin + DPL = Read+Write; NC Admin = None.
  2. Safeguarding Cases → Resolution_Steps → IO Super Admin + DPL only.

Profile-scoped buttons

"Allocate Places" and "Generate Consent Audit Trail" are restricted to a single profile via the button's profile visibility setting — so even though the module is visible to others, only IO Super Admin / DPL see the action.
