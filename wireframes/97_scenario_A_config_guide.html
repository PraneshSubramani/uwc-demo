The permissions foundation

Scenario A configures who can see and do what. It is the base layer every other scenario depends on. This guide reflects the actual demo org: 4 profiles (Administrator, Standard, IO Admissions, NC Reviewer), the role hierarchy + data-sharing rules (which records), blind-review field-level security, and the custom action buttons — step by step in Zoho CRM Setup.

Scope of the demo build (confirmed 30 May 2026)

Profiles in use: Administrator · Standard · IO Admissions · NC Reviewer — only these four.

4 modules deleted from the demo org: Mailing_Lists, Communication_Logs, Consent_Records, Safeguarding_Cases. Communications run through Zoho Campaigns (not Mailchimp, not custom comms modules), so those modules and the Data-Protection-Lead profile they implied are out of scope for the demo.

The one concept that explains everything: Profiles vs Roles

Zoho splits access into two independent systems. Confusing them is the #1 setup error.

PROFILE = what you can DO

Controls which modules, fields and actions a user can touch (View / Create / Edit / Delete). Profile says "you may score applications but not configure programmes."

e.g. IO Admissions can edit allocations; NC Reviewer can only score.

ROLE = which records you SEE

Controls data scope via a hierarchy. A user sees their own records + everyone below them in the tree. Role says "you see your NC's records, not another's."

e.g. an NC Reviewer in Brazil sees Brazil's applications; the same profile in another NC sees theirs.

So record scoping is a ROLE job, not a profile job

Two NC Reviewers can share the identical NC Reviewer profile (same module access) but sit in different roles, so Data Sharing Rules give each only their own NC's records. Build profiles first (Steps 1–2), then roles (Step 3), then layer field security (Step 4) and actions (Step 5).

Prep — modules & admin access

Setup → Modules and Fields
  1. Confirm you are signed in with the Administrator profile. If you hit "Insufficient privileges", you must be promoted to Administrator before any of this works.
  2. Disable unused stock modules (Vendors, Services, Projects, Sales Inbox, stock Questions) via the module kebab → Disable Module.
  3. Delete the 4 out-of-scope modules — Mailing_Lists, Communication_Logs, Consent_Records, Safeguarding_Cases (module kebab → Delete). Comms are handled in Zoho Campaigns.
  4. Leave enabled: Home · Deals (Applications) · Contacts · Programmes · Campaigns · Tasks · Meetings · Calls · plus the custom modules (NCs, Schools, Place Allocations, Nomination Packs, NC Volunteers, Training Records, MoU Obligations, Governance Bodies/Memberships, Competencies).

Note

The "Applications" module is the standard Deals module, relabelled. Permission screens will say Deals.

3

Build the role hierarchy + data-sharing rules

Setup → Users and Control → Roles · then Security Control → Data Sharing Settings

The role tree decides record visibility. Build it top-down, then set the org-wide default to Private so the hierarchy actually restricts.

International Office (top — sees everything) ├── IO Admissions (Fatima — network-wide admissions) ├── NC — Brazil │ ├── NC Reviewer Brazil (Clara) │ └── NC Reviewer Brazil (Paulo) └── NC — Germany / Kenya / India / UK └── NC Reviewer (per NC)
Steps
  1. Under Roles, build the tree — IO Admissions reports to International Office; each NC Reviewer reports to its NC.
  2. Assign uwc_admin the International Office role + Administrator profile.
  3. Go to Security Control → Data Sharing Settings. Set the default access for record-scoped modules (Deals, Contacts) to Private.
  4. With Private default, each role sees only its own records + those below it. A Brazil reviewer can't see Germany's applications — they're sibling branches.
  5. For network-visible modules (Programmes, Schools, Governance), set the default to Public Read Only instead.

This is what enforces NC-level isolation

Profiles give reviewers the same module access. The Private default + sibling roles is what scopes each reviewer to their own NC. Without setting the default to Private, the hierarchy does nothing.

"Own row only" needs a Client Script

Zoho FLS can't scope by row value. Set NC Reviewer to Read+Write at the field level, then add a Client Script on the Application detail page that blanks peer reviewers' cells unless All_Reviews_Complete = true. Script: zoho-build/client-scripts/blind_review.js. This is what makes the blind-review reveal work in Scenario C.

Out of scope for the demo
