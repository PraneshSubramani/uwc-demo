What you're testing

Scenario D moves an applicant (Sofia) into a confirmed school place. The chain is: allocate places → generate a branded pack → nominate to school → school accepts → finance sync fires. Unlike the read-only scenarios, this one exercises 2 custom buttons, 3 workflow rules and 4 Deluge functions across three custom modules — so it's the most worth testing end-to-end.

The whole scenario hinges on one field — Deals.Stage moving Shortlisted → Nominated → Placed. If you watch only one thing, watch that.

Field / config names below are from the build spec — confirm against the live org

API names marked confirm should be verified in the live EU org before relying on them (they can drift from the repo). The four high-value checks: field API names on Deals, the School_Place_Allocations fields incl. Cycle=2026, that the buttons + WR-D-1/2/Pack rules exist, and that the Brazil NC has NC_Code = BR.

Pre-test setup (do once)

1. Run the demo_reset function — resets Sofia to Stage = Shortlisted and Places_Used = 0 on all allocations.
2. Confirm seed data is loaded: 08_place_allocations.csv (15 rows), 02_schools.csv (3 schools), Sofia exists in Deals.

Tests

1 Place Allocation pivot renders any IO profile

Do
Open Place Allocations list → switch to the pivot / matrix view (Group By two dimensions).
Pass
15 cells, schools across the top, NCs down the side. Atlantic ↔ Brazil = "4 / 0" (Allocated / Used).
If it fails
School_Lookup confirm / NC_Lookup confirm / Cycle confirm aren't populated on the rows, or the view isn't grouped by two dimensions.

2 "Allocate Places" button edits a cell IO Super Admin

Do
Click Allocate Places → change Atlantic × Brazil Places_Allocated from 4 → 5 → Confirm.
Pass
Cell shows "5 / 0"; an email goes to Ana.
Verify in data
The Atlantic+Brazil+2026 row now has Places_Allocated = 5.

3 Generate Nomination Pack — the big one NC Admin (Ana)

Do
On Sofia: confirm Stage = Shortlisted, set School_Nominated = UWC Atlantic confirm → Save → click Generate Nomination Pack.
Pass
Toast Pack NP-2026-xxxxx generated. NC Reference: BR-2026-00041. A new Nomination_Packs record appears with Pack_Status = Generated.
Verify the reference logic
The ref is built as NC_Code + "-" + year + "-" + suffix. It needs NC_Code = BR confirm on the Brazil NC and Sofia's Deal_Name to end in -00041. If the ref comes out XX-2026-…, the NC_Code field is empty.
Known soft spot
The function attaches the pack as HTML (PDF needs a Zoho Sign / WorkDrive connection). If the attach fails the record still creates — check the function log for attached N bytes.

4 Preview the pack NC Admin (Ana)

Do
Open the Nomination Pack → click the attached file.
Pass
Navy banner "UWC Nomination Pack — BR-2026-00041", Sofia's name / DOB / country, Brazil NC, UWC Atlantic, personal statement, generated timestamp.

5 Nominate → quota decrements (WR-D-1) NC Admin (Ana)

Do
On Sofia: click Nominate to School → confirm.
Pass — all three must happen
(a) Stage → Nominated · (b) Atlantic ↔ Brazil quota → "5 / 1" (Places_Used++ by decrement_school_quota) · (c) Pack Pack_Status → Sent; email to Helen Richards.
Most likely failure
Quota doesn't decrement → the function couldn't find the allocation. Its search needs School_Lookup, NC_Lookup and Cycle = 2026 all matching. A missing Cycle is the usual culprit.

6 Helen accepts → auto-Placed (WR-D-Pack) School Admissions Director

Do
As Helen: open Nomination Packs (filtered to Sent) → click Sofia's row → click Accept Nomination.
Pass
Pack_Status → Accepted, and Sofia's linked Application auto-flips Stage → Placed (WR-D-Pack firing). Emails to Sofia, Ana, Fatima.
Profile-scoping test
Helen should see only Sofia's pack, not packs for other schools. If she sees others, her profile / data-sharing isn't restricting to her school.

7 Business Central sync (WR-D-2 · stub) IO Super Admin

Do
Open Sofia → check the top BC_Invoice_ID confirm field.
Pass
Populated with INV-2026-{6 digits} (set by send_to_business_central when Stage hit Placed).
Note
This is intentionally a stub — it writes a fake invoice number; the real BC API call is not wired. That's expected and disclosed in the demo. ("If BC is down → queue in Zoho Flow, retry with backoff; Application is marked Placed regardless.")

Re-running the test

decrement_school_quota is not idempotent — each Nominate adds 1 to Places_Used. Before re-testing, run demo_reset (or manually reset Places_Used = 0 and Sofia's Stage = Shortlisted), or the quota will keep climbing.
